# Diataxis Principles Reference

## The Four Modes

Diataxis classifies all documentation into four modes based on the user's need. Every page must serve exactly one mode -- mixing modes within a page is the primary anti-pattern.

### Tutorial (Learning-Oriented)

A tutorial is a **lesson** -- a guided experience where the learner follows a teacher through a series of concrete steps to acquire new skills and understanding.

**Characteristics:**
- First-person plural ("we") affirms the tutor-learner relationship
- Shows the destination upfront: "In this tutorial, we will..."
- Delivers visible results early and often -- every step produces a comprehensible outcome
- Maintains a narrative of the expected -- shows example output, flags likely errors
- Points out what should be noticed: "Notice that..."
- Minimizes explanation ruthlessly -- link to concept pages instead
- Ignores options and alternatives -- stays focused on the path to completion
- Must work for every user, every time (the teacher is absent)

**Structure:**
1. Opening: what we'll accomplish
2. Sequential steps with concrete results at each stage
3. Expected output shown at key checkpoints
4. Closing: what was accomplished, where to go next

**Anti-patterns:**
- "In this tutorial you will learn..." (presumptuous)
- Leading with abstraction or generalization
- Offering choices or alternatives mid-lesson
- Explaining concepts inline instead of linking to explanation pages
- Assuming correctness without testing on real users

### How-To Guide (Task-Oriented)

A how-to guide is a **recipe** -- directions that guide a competent user through solving a specific, real-world problem.

**Characteristics:**
- Written from the user's perspective, not the tool's -- "How do I accomplish X?" not "What can Y do?"
- Goal-oriented and focused -- action and only action
- Assumes the user knows what they want to achieve
- Addresses real-world complexity -- adaptable, not narrowly rigid
- Omits the unnecessary -- practical usability over completeness
- Includes judgment, not just procedure -- solving problems involves thinking
- Describes a logical sequence with flow and rhythm

**Structure:**
1. Title: "How to [accomplish specific goal]"
2. Prerequisites: what the reader needs before starting
3. Steps in logical order with code examples
4. Expected outcome: what success looks like
5. Next steps: related guides or deeper concepts

**Anti-patterns:**
- Tool-focused guidance: "To use the CommandBus..." instead of "How to dispatch a command"
- Teaching or explaining inline -- link to tutorials and concept pages
- Rigid, non-adaptable instructions that only work for one narrow scenario
- Ambiguous titles that don't clarify whether it's how, whether, or what

### Reference (Information-Oriented)

Reference is **austere technical description** -- the facts about the machinery, consulted rather than read.

**Characteristics:**
- Describes and only describes -- neutral, objective, factual
- Adopts standard, consistent patterns -- users know where to find things
- Mirrors the structure of the product -- the documentation map matches the code territory
- Provides examples as concise illustration, not instruction
- Authoritative and unambiguous

**Note:** API reference pages are typically generated by external tools (Sphinx autodoc, mkdocs). This skill focuses on creating concept pages, how-tos, and tutorials. When it touches reference material, it's for structural reference (e.g., configuration options, error codes) rather than API signatures.

**Anti-patterns:**
- Mixing instruction or explanation into reference
- Relying solely on auto-generated docs as complete documentation
- Injecting opinion, speculation, or marketing claims
- Creative or varied writing styles -- consistency is key

### Explanation (Understanding-Oriented)

Explanation is **discursive treatment** of a subject -- providing context, background, and the *why* behind design decisions.

**Characteristics:**
- Makes connections to other things, even outside the immediate topic
- Provides context: design decisions, historical reasons, technical constraints
- Talks *about* the subject -- titles imply "About..."
- Admits opinion and perspective -- considers alternatives and counter-examples
- Kept closely bounded to a conceptual topic, not a task or system component
- Best read away from the product (the only mode you might "read in the bath")

**Structure:**
1. Title: "About [topic]" or "[Topic] explained"
2. Opening: what this page illuminates and why it matters
3. Body: context, history, alternatives considered, design rationale
4. Connections to related concepts
5. Links to relevant ADRs, how-tos, and reference

**Anti-patterns:**
- Scattering explanation in small parcels across other sections
- Letting explanation absorb tutorials or how-tos
- Treating it as a luxury or less important
- Creeping into instruction or technical description

## The Compass: Choosing the Right Mode

The compass uses two binary questions to determine which mode content belongs to:

1. **Action or cognition?** -- Is this about practical steps (doing) or theoretical knowledge (thinking)?
2. **Acquisition or application?** -- Is the user learning/studying or working/applying?

| | Action (doing) | Cognition (thinking) |
|---|---|---|
| **Acquisition (learning)** | **Tutorial** | **Explanation** |
| **Application (applying)** | **How-To Guide** | **Reference** |

### Using the Compass

For new content, ask in this order:
1. What is the user's need? (learn, do, understand, look up)
2. Is this primarily action or cognition?
3. Is the user acquiring new skill or applying existing skill?

For existing content that feels "off":
1. What mode is it currently operating in? (Read the title, opening, and first few paragraphs)
2. What mode should it be in based on the user's need?
3. If they don't match, either rewrite to the correct mode or split into separate pages

### Typical Page Assignments

| Content | Mode |
|---|---|
| "Getting started with X" | Tutorial |
| "How to configure Y" | How-To |
| "X class API" | Reference |
| "Why we chose Z pattern" | Explanation |
| "Understanding the event loop" | Explanation |
| "Install X, then create Y, then deploy Z" | Tutorial |
| "Error code reference" | Reference |

## Page Titling Conventions

| Mode | Title Pattern | Examples |
|---|---|---|
| Tutorial | "[Topic] tutorial" or "Getting started with [X]" | "Shopping cart tutorial", "Getting started with aggregates" |
| How-To | "How to [verb phrase]" | "How to define an aggregate root", "How to configure the command bus" |
| Reference | "[Component] reference" or "[API] specification" | "CommandBus API reference", "Configuration options" |
| Explanation | "[Topic]" or "About [topic]" or "Understanding [topic]" | "Event sourcing concepts", "Understanding the Unit of Work pattern" |

## Mode Discipline Checklist

When reviewing or writing a page, verify:
- [ ] The title matches the mode (How to... for how-tos, not for explanations)
- [ ] The opening paragraph signals the right mode ("This guide shows you how to..." vs "This section explains the design behind...")
- [ ] No teaching language in how-tos ("you will learn", "first, let's understand")
- [ ] No instructional language in reference ("to use this, call...")
- [ ] No scattered explanation in tutorials (link to concept pages instead)
- [ ] Cross-references point to the right mode (tutorials link to concepts; how-tos link to reference and concepts)
